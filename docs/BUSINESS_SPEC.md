### 2-2. Event Catalog (Consolidated)

> All OCPP messages (except Heartbeat) are converted to domain messages and published to message brokers.
> This section documents every event with explicit Publisher → Consumer mappings across all transports.

**Transport Selection Policy**:
- **RabbitMQ** — Messages requiring **time-ordering + persistence** (must not be lost). Examples: billing, payment, session lifecycle events.
- **Redis Streams/Pub/Sub** — Simple event delivery where occasional loss is tolerable or ordering is non-critical. Examples: commands, status updates, real-time telemetry.

#### F. Message Ordering Guarantees

| Transport | Scope | Ordering | Mechanism |
|-----------|-------|----------|-----------|
| RabbitMQ → domain-logic-processor | **Per-charger** | **Guaranteed** | FNV-1a hash(`chargerID`) % workerCount → same worker (sequential) |
| RabbitMQ → domain-logic-processor | Cross-charger | Not guaranteed | Different chargers dispatched to different workers (concurrent) |
| Redis Streams (commands) | Per-stream | FIFO | XReadGroup batch=100, sequential processing per batch |
| Redis Streams (domain events) | Per-stream | FIFO (single instance) | Multi-instance: consumer group distributes → no cross-instance ordering |
| Redis Pub/Sub | None | Not guaranteed | At-most-once, fire-and-forget |

> **Why per-charger ordering matters**: Billing and payment events (SESSION_STARTED → SESSION_ENDED → INVOICE_GENERATED → PAYMENT.RESULT) must be processed in order for the same charger. domain-logic-processor's `ScalableWorkerPool` (10–100 workers, `worker/scalable_pool.go`) guarantees this via deterministic hash routing. Events for different chargers are safely processed in parallel.
>
> **RabbitMQ queue**: Single shared queue `domain-logic-processor.events` with prefetch=10. All 5 category bindings (CHARGING, PAYMENT, STATUS, CONNECTION, MAINTENANCE) deliver to the same queue. Worker pool dequeues and hash-routes by `DeviceID`.
>
> **Redis Streams caveat**: `ocpp:commands` and `domain:events:*` streams are global (not partitioned by charger). When running multiple consumer instances (e.g., 100 charger-gateway instances), consumer group distributes messages across instances — no per-charger affinity at this level.

## 3. State Transition Definitions

### 3-1. Charging Session States

| From | To | Condition | Trigger | Redis Key + TTL |
|------|----|-----------|---------|-----------------|
| (none) | AUTH_AUTHORIZING | RFID/QR auth request received | charger-authorizer: AuthorizeCharger / AuthorizeRFID | `session:{id}` TTL 15min |
| AUTH_AUTHORIZING | AUTH_AUTHORIZED | Auth success (valid card + no overdue) | charger-authorizer | `session:{id}` TTL 15min maintained |
| AUTH_AUTHORIZING | AUTH_REJECTED | Auth failure (invalid card, delinquent, brute-force) | charger-authorizer | Session key deleted |
| AUTH_AUTHORIZED | CHARGING_ACTIVE | StartTransaction / TransactionEvent(Started) received | charger-authorizer: NotifyChargingStarted | `session:{id}` TTL 24h |
| AUTH_AUTHORIZED | EXPIRED | 15min TTL exceeded (no StartTransaction) | Redis TTL expiry | Auto-deleted |
| CHARGING_ACTIVE | COMPLETED | StopTransaction / TransactionEvent(Ended) received | charger-authorizer: NotifyChargingStopped | `session:{id}` TTL 1h (retention) |
| CHARGING_ACTIVE | EXPIRED | 24h TTL exceeded (abnormal) | OrphanSessionDetector (30min cycle) | 0 KRW Invoice + session EXPIRED |

**Prohibited Transitions**: AUTH_REJECTED → CHARGING_ACTIVE, COMPLETED → CHARGING_ACTIVE, EXPIRED → CHARGING_ACTIVE

### 3-2. Invoice States

| From | To | Condition | Trigger |
|------|----|-----------|---------|
| (none) | PENDING | Invoice created | domain-logic-processor: TariffEngine |
| PENDING | PAID | ExecutePayment success | domain-logic-processor: PaymentResultConsumer |
| PENDING | UNPAID | Payment final failure (FINAL_FAILED) | domain-logic-processor: PaymentResultConsumer |
| PENDING | PAID | Charger built-in payment confirmed (CHARGER_PAYMENT) | domain-logic-processor: PAYMENT.CONFIRM consumed |
| PENDING | ROAMING_SETTLED | Roaming user (PG call skipped, delegated to eMSP settlement) | domain-logic-processor: IsRoamingUser() = true |
| PAID | REFUNDED | CPO admin refund processing complete | domain-logic-processor: ProcessRefund → CancelPayment success |
| UNPAID | PAID | OverdueChecker retry success | domain-logic-processor: OverdueChecker (60min cycle) |
| UNPAID | WRITTEN_OFF | All 5 retries failed → write-off | domain-logic-processor: MarkInvoiceWrittenOff |

**Prohibited Transitions**: PAID → UNPAID, REFUNDED → PAID, WRITTEN_OFF → PAID, ROAMING_SETTLED → PAID

> **ROAMING_SETTLED**: Roaming invoices are delegated to eMSP settlement without PG payment. cpo-settlement aggregates separately via the roaming_settlements table. Separated from PAID to clearly distinguish normal vs. roaming in settlement reports.
>
> **WRITTEN_OFF Recovery**: WRITTEN_OFF is a terminal invoice state. When a written-off user manually pays via customer service, the CS agent clears the `unpaid:{user_id}` Redis flag and marks the `unpaid_record` as RESOLVED via task-supporter. The invoice itself remains WRITTEN_OFF (settlement reports already recorded the bad_debt). A new receipt is created referencing the manual payment.

### 3-3. blockchain_status (STO)

| From | To | Condition | Trigger |
|------|----|-----------|---------|
| (none) | PENDING | PAID.INVOICE_CONFIRMED event consumed | blockchain-bridge: Redis Streams consumer |
| PENDING | CONFIRMED | processCharge() receipt confirmed + NFT minted | blockchain-bridge: EnergyFi L1 RPC |
| PENDING | FAILED | Max retries exceeded (default: 5) | blockchain-bridge: scheduler |
| FAILED | PENDING | Operator manual reprocessing | platform-console: manual trigger |

### 3-4. Write-Off (WRITTEN_OFF) Post-Processing

- **Settlement Impact**: During cpo-settlement monthly processing, WRITTEN_OFF invoices are aggregated separately as `bad_debt` line items. Not deducted from CPO settlement (absorbed as platform loss).
- **User Service Restoration**: The `unpaid:{user_id}` Redis flag is retained at write-off. When the user manually pays the overdue amount through customer service, the CS agent clears the Redis flag and marks the `unpaid_record` as RESOLVED via task-supporter. The invoice remains WRITTEN_OFF (already recorded in settlement). A new manual payment receipt is created separately.
- **Statistics**: statistics-builder monthly reports include bad_debt_count and bad_debt_amount fields.

### 3-5. Payment Validation Policy

Payment validation is NOT performed by the platform at session start. No pre-authorization (hold) occurs. The rationale:

| Auth Method | Who Pays | Validation Point | Platform Action |
|-------------|----------|-------------------|-----------------|
| **Mobile App** | User (PG card) | Mobile backend validates registered payment card BEFORE sending RemoteStart | None — already validated upstream |
| **RFID Card** | MOE (government) | MOE member card status checked by charger-authorizer (valid/blocked) | Platform cannot pre-auth government settlement — only card status check |
| **OCPI Roaming** | eMSP (roaming partner) | eMSP Authorize token validated by charger-authorizer | Settlement delegated to eMSP — no platform PG involvement |

**Key design decisions:**
- `CHARGING.SESSION_STARTED` → create session record only (no payment hold)
- `CHARGING.SESSION_ENDED` → TariffEngine calculation → `ExecutePayment` (direct payment, not capture)
- If payment fails → invoice status UNPAID → OverdueChecker retries (5x, 60min interval) → WRITTEN_OFF
- Zero-energy sessions (0 Wh) → session COMPLETED, no invoice created

```

    ■ OCPP 1.6:
    → [charger-authorizer] NotifyChargingStarted (session: CHARGING_ACTIVE, 24h TTL)

    ■ OCPP 2.1:
    → [charger-authorizer] NotifyChargingStarted (session creation + CHARGING_ACTIVE transition handled simultaneously, 24h TTL)

    ↓
[domain-logic-processor] Session started processing:
    • Parse event → validate charger exists (GetChargerByDeviceID)

    ↓

    → [charger-gateway / v21] OCPP MeterValues → Redis cache (latest value) + TimescaleDB (raw OCPP message archival only)

    ↓
```

### C. 3-Way Billing Mode (Tariff Calculation + Payment)

Three payment paths based on charger type.

> **Amount Unit Rule**: TariffEngine internal calculations use "cents" (1/100 of KRW). Invoice DB storage and PG payments use KRW (`centsToKRW`: rounded `(cents+50)/100`). ExecutePayment amounts are in KRW.

#### C-1. SERVER_SIDE (Server Tariff Calculation)

Standard path where the server calculates the tariff and processes payment.

```
SESSION_ENDED event received
    ↓
1. Charger info query: charger-station-manager → billing_mode, country_code, moe_registered
2. Tariff resolution (3-tier fallback):
    Station tariff (DB) → MOE regulated tariff → Default tariff (per-country fallback: KR/US/DE/JP)
3. Tariff calculation (internal cents):
    Energy cost = kWh × unit price × 100
    Time cost = minutes × unit price × 100 (optional)
    (Environmental levy 2%, Electricity fund 3.7%: CPO pays separately — not included in customer rate)
    VPP/REC revenue adjustment (seasonal)
4. Invoice creation (cents → KRW conversion, then DB save)
5. Payment:
    ExecutePayment(user_id, amount KRW) → direct payment via NICE Pay
6. Receipt creation → Invoice PAID
```

#### C-2. CHARGER_INVOICE (Charger Invoice, Server Validation)

Charger calculates the tariff; server validates before payment.

```
SESSION_ENDED → EventCache save (10min TTL)
INVOICE_GENERATED → Pair with SESSION_ENDED from EventCache
    ↓
Server recalculates to validate charger amount
    • Tolerance: ±5% or ±1,000 KRW (accept if either condition is satisfied)
    • Within tolerance → use charger amount
    • Discrepancy → record in invoice_discrepancies → use server amount
    • alert-messenger: Slack alert on discrepancy (repeated occurrences trigger DERA analysis)
    ↓
Payment execution (same as SERVER_SIDE)
```

#### C-3. CHARGER_PAYMENT (Charger Built-in Payment Terminal)

Charger handles payment directly. Server records only.

```
SESSION_ENDED → EventCache save
PAYMENT.CONFIRM → Pair with SESSION_ENDED from EventCache
    ↓
Trust charger payment result:
    • Invoice creation (status=PAID)
    • Receipt creation (charger transaction_id)
    • No server-side PG call
```

**Participating Services**: domain-logic-processor → payments-gateway → charger-station-manager (tariff query)

### D. Payment Failure → Overdue Collection

Collection cycle when payment finally fails.

```
[payments-gateway] PAYMENT.RESULT(FINAL_FAILED) event published
    ↓
[domain-logic-processor] PaymentResultConsumer:
    • Invoice → UNPAID
    • unpaid_record created (retry_count=0)
    • Redis SET unpaid:{user_id} (30-day TTL)
    • Push notification: "You have an overdue charge"
    ↓
[charger-authorizer] Next charging attempt:
    • Redis GET unpaid:{user_id} → DELINQUENT rejection
    ↓
[domain-logic-processor] OverdueChecker (60min cycle):
    • Query unpaid_records (retry_count < 5)
    • ExecutePayment() retry
    • Success → delete unpaid_record, clear Redis flag, Invoice PAID
    • Failure → increment retry_count
    ↓
All 5 attempts failed → OverdueChecker auto write-off (MarkInvoiceWrittenOff)
    • Invoice → WRITTEN_OFF
    • unpaid:{user_id} Redis flag retained (service block continues)
    • Settlement: aggregated as bad_debt line item (→ Section 3-4)
    • alert-messenger: Slack alert for write-off
```

**Separate Schedulers**:
- **PendingInvoiceRetryer** (10min): Retries PENDING invoices exceeding 15min (PG outage recovery)
- **OrphanSessionDetector** (30min): Sessions ACTIVE 24h+ → 0 KRW Invoice + session EXPIRED

**Participating Services**: domain-logic-processor → payments-gateway → charger-authorizer

### P. STO Revenue Distribution (Blockchain Bridge)

6-stage pipeline for recording charging transactions as ERC-721 Soulbound NFTs on EnergyFi L1 via ChargeRouter contract.

```
[payments-gateway] Payment complete (Invoice PAID)
    → Redis Streams: domain:events:paid (PAID.INVOICE_CONFIRMED)
    ↓
[blockchain-bridge] Stage 1 — Event consumption & lookup:
    • Consume from Redis Streams (consumer group: blockchain-bridge)
    • Query business.invoices + business.charging_sessions → get session data
    • Invoice not found or zero amount → ACK and skip
    ↓
Stage 2 — Record & save:
    • Build RevenueDistribution: sessionId, chargerId, stationId, energyKwh, distributableKrw, regionCode
    • INSERT ledger.sto_revenue_distributions (blockchain_status=PENDING)
    • Deduplicate by session_id (UNIQUE constraint)
    ↓
Stage 3 — On-chain submission:
    • ChargeRouter.processCharge(invoiceId, sessionId, chargerId, stationId, energyKwh, distributableKrw, regionCode, seSignature)
    • Non-payable (no AVAX/ETH value transfer)
    • eth_sendRawTransaction → EnergyFi L1 (chainId: 31337)
    ↓
Stage 4 — Confirmation:
    • eth_getTransactionReceipt polling (every 2s, max 30s)
    • receipt.status = 0x1 → success
    • Decode ChargeProcessed event → extract tokenId (ERC-721 NFT)
    ↓
Stage 5 — DB recording:
    • sto_revenue_distributions updated (tx_hash, block_number, token_id, confirmed_at)
    • blockchain_status → CONFIRMED
    ↓
Stage 6 — NFT record:
    • Charging record permanently stored as Soulbound NFT on-chain
    • Non-transferable — immutable proof of charging transaction

Retry policy (background scheduler):
    • Runs every 5 minutes, picks up PENDING records older than 5 minutes
    • Max retries: 5 (configurable)
    • After max retries: blockchain_status=FAILED, operator alert
```

**Participating Services**: payments-gateway → Redis Streams → blockchain-bridge → EnergyFi L1
**Contract**: ChargeRouter (single contract, ERC-721 Soulbound NFT)
