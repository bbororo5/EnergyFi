# ERC-165 Application Note

## 0. Summary

- Directly visible application: `ChargeTransaction`
- Indirect application: upgradeable contracts inheriting `AccessControlUpgradeable`
- Current role: standard on-chain interface detection

## 1. Purpose and Background

ERC-165 standardizes how a contract declares which interfaces it supports.

Its core function is:

- `supportsInterface(bytes4 interfaceId) returns (bool)`

That makes it easier for contracts and tools to detect capabilities without guessing or relying entirely on off-chain ABI knowledge.

## 2. Why It Matters for EnergyFi

EnergyFi is a multi-contract system. Standard interface detection helps with:

- tooling and indexer integration
- contract introspection after upgrades
- recognizing standard surfaces such as ERC-721 and access-control interfaces

It is not a business-domain standard by itself, but it improves integration reliability.

## 3. Standard Content

ERC-165 defines three key ideas:

- a contract can expose `supportsInterface(bytes4)`
- interface IDs are derived from XORed function selectors
- the check must remain lightweight

Common interface IDs include:

- `IERC165`: `0x01ffc9a7`
- `IERC721`: `0x80ac58cd`
- `IERC721Metadata`: `0x5b5e139f`

## 4. Where EnergyFi Applies It

### 4.1 Explicit override

`ChargeTransaction` explicitly resolves `supportsInterface()` because it combines multiple parents that expose interface-detection logic.

### 4.2 Indirect inheritance

Contracts inheriting `AccessControlUpgradeable` gain ERC-165-based support for access-control interface detection.

Representative examples:

- `DeviceRegistry`
- `StationRegistry`
- `ChargeRouter`
- `RevenueTracker`
- `ReputationRegistry`
- `RegionSTO`
- `RegionSTOFactory`

## 5. EnergyFi-Specific Interpretation

EnergyFi uses ERC-165 mainly for standard surfaces:

- ERC-721-related discovery
- access-control discovery

Project-specific interfaces such as `IChargeTransaction` or `IRevenueTracker` are not automatically discoverable through ERC-165 unless the implementation explicitly advertises them.

## 6. Cautions

- ERC-20 is not an ERC-165-native standard, so ERC-20 support should not be inferred through `supportsInterface()`
- ERC-165 helps identify standardized interfaces, but it is not a full replacement for repository-controlled address maps and ABI knowledge

## 7. Practical Takeaway

In EnergyFi, ERC-165 is an integration standard:

- useful for tooling
- useful for post-upgrade checks
- not itself a domain-specific business primitive
