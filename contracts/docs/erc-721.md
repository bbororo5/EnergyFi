# ERC-721 Application Note

## 0. Summary

- Applied contract: `ChargeTransaction`
- Base implementation: OpenZeppelin `ERC721Upgradeable`
- Current role: unique on-chain proof record per settled charging session
- Important qualification: this is not a collectible NFT design; it is a session-evidence model

## 1. Purpose and Background

ERC-721 standardizes unique, non-fungible tokens.

It exists for cases where each record must remain individually identifiable, such as:

- certificates
- deeds
- serial-numbered assets
- unique rights or evidence records

The standard defines a common surface for token ownership, transfer, approval, and optional metadata.

## 2. Why It Matters for EnergyFi

Each charging session is unique because it carries its own:

- `sessionId`
- charger and station context
- timestamps
- energy amount
- distributable revenue
- SE signature

If EnergyFi collapsed sessions into a fungible model, it would lose the audit trail that later revenue and carbon pipelines depend on. ERC-721 is therefore used as a uniqueness model for session records.

## 3. Standard Content

Core ERC-721 functions include:

- `balanceOf(address)`
- `ownerOf(uint256)`
- `approve(address,uint256)`
- `getApproved(uint256)`
- `setApprovalForAll(address,bool)`
- `isApprovedForAll(address,address)`
- `transferFrom(address,address,uint256)`
- `safeTransferFrom(...)`

Common metadata functions include:

- `name()`
- `symbol()`
- `tokenURI(uint256)`

## 5. EnergyFi-Specific Interpretation

### 5.1 One session, one token

`ChargeTransaction.mint()` creates exactly one token for one settled charging session after validation.

Typical checks include:

- duplicate session rejection
- station registration validation
- chip activation validation
- SE signature verification

### 5.2 Soulbound behavior

The token is not meant to move between user wallets.

The current design uses ERC-721 for uniqueness, while blocking post-mint transfer behavior. In practice, the token behaves like an immovable audit artifact rather than a tradable NFT.

### 5.3 Metadata is not the primary data source

EnergyFi relies on:

- contract storage
- structured getters
- emitted events

rather than marketplace-style metadata as the main evidence surface.

## 6. Cautions

- external NFT tools may expect normal wallet ownership and transferable behavior
- `ChargeTransaction` behaves like a Soulbound token but does not currently claim ERC-5192 support
- enumerable NFT assumptions should not be projected onto the contract unless the code explicitly provides project-specific iteration helpers

## 7. Practical Takeaway

In EnergyFi, ERC-721 means:

- unique session identity
- auditable per-session evidence
- the root dataset for revenue and carbon derivation

It does not mean a collectible-NFT user experience.
