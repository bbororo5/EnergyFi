# ERC-1967 Application Note

## 0. Summary

- Applied implementation: `EnergyFiProxy`
- Base implementation: OpenZeppelin `ERC1967Proxy`
- Current role: standardize proxy storage slots for EnergyFi's upgradeable contracts

## 1. Purpose and Background

ERC-1967 exists to standardize where a proxy stores critical addresses such as:

- the implementation address
- the admin address
- the beacon address

That reduces ambiguity for deployment tooling, upgrade tooling, and auditors, and helps avoid storage-slot collisions between proxy state and implementation state.

## 2. Why It Matters for EnergyFi

EnergyFi uses proxies for multiple long-lived contracts. A standard proxy-slot convention matters because it allows:

- consistent deployment tooling
- auditable implementation discovery
- compatibility with established OpenZeppelin upgrade infrastructure

## 3. Standard Content

ERC-1967 defines standard slots and related events such as:

- `Upgraded(address implementation)`
- `AdminChanged(address previousAdmin, address newAdmin)`
- `BeaconUpgraded(address beacon)`

It does **not** define the full upgrade-authority model by itself. That is why EnergyFi pairs it with UUPS / ERC-1822.

## 4. Where EnergyFi Applies It

Primary file:

- `contracts/contracts/utils/ERC1967ProxyImport.sol`

Representative usage points:

- deployment scripts under `contracts/scripts/`
- test helpers for proxy deployment
- factory-driven proxy deployment flows

## 5. EnergyFi-Specific Interpretation

### 5.1 Proxy storage convention, not business logic

ERC-1967 matters in EnergyFi because it defines how upgradeable deployments are stored and inspected, not because it changes charging, revenue, or STO business behavior.

### 5.2 Works together with UUPS

In EnergyFi's current architecture:

- ERC-1967 defines where the proxy stores implementation information
- UUPS / ERC-1822 defines how upgrades are authorized and compatibility-checked

## 6. Cautions

- any custom proxy wrapper behavior must still preserve the ERC-1967 storage convention
- deployment flows that allow delayed initialization must be handled carefully to avoid uninitialized proxy risk

## 7. Practical Takeaway

In EnergyFi, ERC-1967 is the storage-convention layer of the upgrade architecture. It is what makes upgradeable proxy deployments inspectable and consistent across tooling.
